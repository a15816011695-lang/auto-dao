# 难度适配设计：前置知识检测 + 大测驗体系

**日期**: 2026-04-15
**状态**: 设计已确认，待实施
**作者**: Claude Code (brainstorming with 用户)

---

## 背景与目标

### 问题

auto-tutor 生成的课程难度超出用户实际水平。用户反馈：生成的 lesson 内容标注为"理解+应用（Bloom L2-L3）"，但实际内容包含 L4 分析层级的知识点。

根本原因：**前置知识缺口未被检测**。用户跳过了基础章节直接进入通信总线学习，系统没有识别出缺失的前置概念（如"时钟信号"），导致 CPOL/CPHA 等依赖前置知识的课程变成了"从天而降"的概念。

### 目标

建立双层难度适配体系：
1. **课前诊断**：在每课生成前检测前置知识缺口，动态插入补救课
2. **大测驗**：一次性全面评估用户在某专题上的完整知识水平，建立知识图谱

---

## 整体架构

### 双层体系概览

```
大测驗（一次性，100+题）          课前诊断（每课前，3-8题）
    ↓                                  ↓
建立全局知识图谱  ←──────────────→  微调局部前置缺口
    ↓
所有后续 Lesson 生成均基于知识图谱
```

**关系**：大测驗建立全局基线，课前诊断做局部微调。两者互补，不是替代关系。

### 课前诊断插入位置

在现有 pipeline 的 **Step 3（上下文恢复）末尾**、**Step 4（生成主课 Lesson）之前**，新增 Step 3.5：

```
Step 3（上下文恢复）
    ↓
Step 3.5 【课前诊断】 ← 新增
    ↓
Step 4（主课 Lesson 生成）
```

### Step 3.5 完整流程（含闭环）

```
3.5.1 分析下一课所需前置知识（自动 + 手动，递归依赖链展开）
         ↓
3.5.2 生成诊断小测（快速/完整模式，用户选择）
         ↓
3.5.3 用户答题（或选择跳过）
         ↓
3.5.4 批改 + 判定
         ├ 掌握度 < 33% → 插入前置补救课(s)
         ├ 33% ≤ 掌握度 < 60% → 主课开头快速回顾
         └ 掌握度 ≥ 60% → 直接进入主课
         ↓
3.5.5 补救课完成 → 重新快速诊断 → 通过后才进入主课 ← 【闭环】
         ↓
3.5.6 更新用户背景（薄弱知识点 + 诊断记录）
```

## 参数表

| 参数 | 值 | 理由 |
|------|-----|------|
| 前置知识标注 | 自动分析 + 手动补充兜底 | 自动发现大部分缺口，手动补充边界情况 |
| 小测模式 | 快速模式(3题/3分钟) / 完整模式(8题/10分钟) | 用户可选 |
| 跳过选项 | 允许跳过，记录"未诊断" | 不强制，但留下痕迹 |
| 掌握标准 | 60% 正确率 | 发现问题导向，宁宽勿严 |
| 触发补救课 | 掌握度 < 33%（即错 ≥ 2 题） | 配合 60% 标准，2 题错已是明显信号 |
| 触发快速回顾 | 33% ≤ 掌握度 < 60%（即错 1 题） | 轻量处理，不单独开课 |
| 知识判定原则 | 答案答偏但知识点对 → 算会（部分正确） | 三档评估，部分正确视同掌握 |
| 补救课完成后 | 重新快速诊断（3题），通过后才进入主课 | 确保前置缺口真正填补 |
| 依赖链深度 | 递归展开直到叶子前置知识点 | 防止多层依赖链断裂 |

## 详细设计

### 3.5.1 前置知识自动分析（含递归依赖链）

#### 核心思路

在生成下一课 Lesson 之前，系统自动扫描下一课的内容，找出"陌生术语"——即**在本课讲解中出现、但在所有已学课程和知识图谱中从未出现过**的概念。同时支持递归展开多层依赖链。

#### prerequisite-map.md 格式（支持递归依赖）

```markdown
# 前置知识依赖表

> 用户可手动维护，支持多级递归依赖。
> 格式：`知识点 | 前置依赖1, 前置依赖2, ...`
> 系统会自动递归展开所有依赖链直到叶子节点。

| 知识点 | 前置依赖 |
|--------|---------|
| 时钟信号 | 数字电路基础 |
| 数字电路基础 | 电平概念（高电平/低电平） |
| SPI CPOL/CPHA | 时钟信号, 串行通信基础 |
| CAN 总线 | SPI 协议原理 |
| 嵌入式操作系统 | C 语言指针, 中断机制 |
| DMA 传输 | 总线架构基础 |
```

#### 递归依赖展开算法

当检测到前置知识点 X 时，检查 X 本身是否也有前置依赖。若有，递归展开直到叶子节点：

```
下一课：SPI 协议原理
    ↓ 检测到前置缺口：时钟信号
    ↓ 检查 prerequisite-map：时钟信号 → 数字电路基础
    ↓ 数字电路基础 → 电平概念
    ↓ 电平概念（叶子节点，无更早依赖）
    ↓
最终补救课列表：
  1. lesson_4_prereq_electric_level.md（电平概念）
  2. lesson_4_prereq_digital_circuit.md（数字电路基础）
  3. lesson_4_prereq_clock_signal.md（时钟信号）
```

#### 自动分析算法

1. 读取 `roadmap_status.md`，提取所有已标记为"✅ 已完成"的知识点
2. 读取知识图谱（`background.md` 中大测驗结果），获取已掌握的知识点
3. 读取下一课的讲解内容草稿，提取关键术语
4. 比对：哪些术语在已学课程/知识图谱中从未覆盖？
5. 对每个缺口，递归查询 `prerequisite-map.md` 展开依赖链
6. 去重 + 排序（叶子前置先补，主课后补）

#### 输出格式

```json
{
  "next_lesson": "SPI 协议原理",
  "prerequisite_gaps_flat": [
    { "concept": "电平概念", "depth": 2, "source": "recursive: 时钟信号 → 数字电路基础" },
    { "concept": "数字电路基础", "depth": 1, "source": "recursive: 时钟信号" },
    { "concept": "时钟信号", "depth": 0, "source": "auto-detect" }
  ],
  "remediation_order": ["电平概念", "数字电路基础", "时钟信号"]
}
```

### 3.5.2 诊断小测生成

#### 两种模式

| 维度 | 快速模式 | 完整模式 |
|------|---------|---------|
| 题数 | 3 题 | 8 题 |
| 覆盖策略 | 优先覆盖优先级最高的前置概念 | 覆盖全部检测到的前置知识缺口 |
| 预计耗时 | 3 分钟 | 10 分钟 |
| 题型 | 每题 1 问 + 1 行填空 | 混合：选择/填空/简答 |

#### 快速模式选题策略（优先级排序）

当前置缺口数量 > 3 时，按以下优先级选择 3 题：

1. **核心依赖最强**：在主课中出现频率最高的前置概念（影响面最大）
2. **历史薄弱点**：该知识点在 `background.md` 的"薄弱知识点"历史记录中出现
3. **依赖链深度最深**：递归链中层级最深的（叶子节点的前置前置）

#### 题目生成逻辑

题目从下一课的学习资料中**反向提取**——从资料的讲解段落中提炼出一个"前置问题"：

```
前置概念：时钟信号
    ↓
生成题型（从资料中反向提取）：
    ① 选择题：什么是时钟信号在通信中的作用？（A. 供电 B. 同步数据 C. 复位）
    ② 填空题：时钟信号的_____决定了数据的采样时刻
    ③ 简答题：为什么通信协议需要时钟信号？举一个身边的例子说明
```

#### 诊断模式选择交互（非阻塞性询问）

在 Step 3.5 开始时，系统非阻塞性地询问：

```
📋 Lesson {N} 课前诊断

请选择诊断模式：
- 快速模式（3题，约3分钟）— 适合时间紧张或复诊
- 完整模式（8题，约10分钟）— 适合首次学习新专题
- 直接跳过（不推荐）— 有前置知识缺口风险

若未选择，默认使用完整模式。
```

用户也可主动触发大测驗：`"我想做个全面测试"` 或 `"大测驗"`。

#### 诊断文件格式

**文件名**：`lessons/prereq_diagnostic.md`

```markdown
# 课前诊断 — Lesson {N}: {lesson_title}

> 本诊断用于检测学习本课所需的前置知识掌握情况。
> 预计耗时：快速模式约 3 分钟，完整模式约 10 分钟。

**诊断模式**：[快速 / 完整]
**前置知识缺口**：{concept_1}、{concept_2}、{concept_3}

---

## 开始诊断

> 请在每道题的"我的答案"区域填写答案。

### 诊断题 1：{concept_name}

{question_content}

**前置知识点**：{related_concept}
**来源提示**：{source_hint}

**我的答案**：

---
```

#### 对话通知格式

```
📋 Lesson {N} 课前诊断已生成！

📄 文件位置：`./learning-history/{topic}_{timestamp}/lessons/prereq_diagnostic.md`

⏱️ 预计耗时：{3分钟 / 10分钟}

📊 前置知识缺口检测到 {N} 个：
  - {concept_1}
  - {concept_2}
  - ...

请阅读诊断文件，完成后告诉我"诊断完成"。
```

⛔ **BLOCKING**：等待用户说"诊断完成"或选择跳过。

### 3.5.4 批改与判定

#### 三档评估在前置小测中的应用

沿用 Step 4.3 的评估框架：

| 评估结果 | 判定标准 | 在前置小测中的含义 | 分数 |
|---------|---------|------------------|------|
| **✅ 完全正确** | 结论对 + 推理完整 | 该前置知识点已掌握 | 1.0 |
| **⚠️ 部分正确** | 结论对但推理有瑕疵，或答偏但知识点实质正确 | 该前置知识点**基本掌握** | 0.5 |
| **❌ 错误** | 结论错误或完全不知道 | 该前置知识点**未掌握** | 0.0 |

#### "答偏但知识点对"的判定规则

**核心原则**：如果用户的答案能让一个了解该领域的人点头说"他其实理解了这个概念，只是没按标准答案的方式说"，就算部分正确。

判定流程：
1. **提取答案中的核心概念**：用户答案里提到了什么知识点？
2. **比对题目考察的知识目标**：这道题希望用户掌握什么？
3. **判断是否在同一个知识体系内**：
   - ✅ 部分正确：用户答的是相关概念，逻辑合理，只是表达或角度不同
     - 例：题目问"时钟信号的作用"，用户答"同步数据的节奏，类似指挥家打拍子" → 知识点完全正确，只是用了类比而非定义
   - ❌ 错误：用户答的是无关概念，或逻辑明显错误
     - 例：题目问"时钟信号的作用"，用户答"给设备供电" → 完全错误

#### 掌握度计算

```
掌握度 = (完全正确题数 × 1.0 + 部分正确题数 × 0.5) / 总题数 × 100%
```

示例：
- 快速模式 3 题：2 题完全正确 + 1 题错误 = (2 + 0) / 3 = **66.7%** ≥ 60% → 通过
- 完整模式 8 题：4 题完全正确 + 2 题部分正确 + 2 题错误 = (4 + 1) / 8 = **62.5%** ≥ 60% → 通过
- 完整模式 8 题：3 题完全正确 + 1 题部分正确 + 4 题错误 = (3 + 0.5) / 8 = **43.75%** < 60% → 触发补救课

#### 判定结果与路由

| 掌握度 | 结果 | 行动 |
|--------|------|------|
| ≥ 60% | ✅ 通过诊断 | 主课直接生成，无需额外铺垫 |
| 33% ≤ x < 60% | ⚠️ 部分通过 | 主课生成，Lesson 文件开头增加"快速回顾"小节 |
| < 33% | ❌ 未通过 | 插入前置补救课，主课延后 |

#### 批改输出格式

```markdown
---

## 诊断结果

| 诊断题 | 考察知识点 | 评估 | 说明 |
|--------|-----------|------|------|
| 诊断题 1 | 时钟信号 | ✅ 完全正确 | 概念清晰 |
| 诊断题 2 | 采样沿 | ⚠️ 部分正确 | 答偏，但知道采样相关 |
| 诊断题 3 | SCK空闲电平 | ❌ 错误 | 概念混淆 |

**掌握度**：{M/N} 题正确（{percentage}%）

**结论**：⚠️ 部分通过 → 主课 Lesson {N} 开头增加快速回顾
```

#### Bloom 层级动态下调（与 4.4 动态支架联动）

若连续 2 次诊断都触发了补救课，说明用户基础偏弱，系统自动降低后续所有课的**目标认知层级**：

| 连续补救课次数 | 目标 Bloom 层级下调 |
|--------------|-------------------|
| 2 次 | L4 → L3（L3+应用 → L2理解+基础应用） |
| 3 次 | L3 → L2（L2理解 → L1记忆+基础理解） |

此下调**不影响已学过的课程**，只对后续新课生效。下调信息记录在 `background.md` 的"备注"区域。

### 3.5.5 补救课完成 → 重新诊断（闭环）

这是确保前置缺口真正填补的关键步骤。

#### 流程

```
插入前置补救课(s)
    ↓ 用户完成补救课
重新生成快速诊断（3题，针对刚补的薄弱知识点）
    ↓
批改快速诊断
    ├ 通过（≥ 60%）→ 进入主课 Lesson
    └ 仍不通过（< 60%）
         ├ 若该知识点已连续补过 2 次 → 降低目标 Bloom 层级 + 进入主课（主课加大铺垫）
         └ 若 < 2 次 → 再补一次更精简的补救课
```

#### 重新诊断文件格式

文件名：`lessons/prereq_diagnostic_recheck.md`

格式同 `prereq_diagnostic.md`，但标注为"复诊"：

```markdown
# 课前诊断（复诊）— Lesson {N}: {lesson_title}

> 复诊目的：验证前置补救课（{gap_name}）是否已真正掌握。
> 预计耗时：3 分钟（快速模式 3 题）

**原诊断掌握度**：{X}%
**已完成的补救课**：{gap_1}、{gap_2}、...
```

#### 对后续课程生成的影响

Lesson 生成时，系统会：
1. 读取 `background.md` 的"薄弱知识点"区域
2. 在讲解内容中，若涉及薄弱知识点，自动在开头加上快速回顾小节
3. 讲解深度和用词参考"备注"区域的教学建议
4. 目标 Bloom 层级参考"备注"中的下调记录

## 前置补救课格式

### 设计原则

前置补救课和普通 Lesson 的定位不同：

| 维度 | 普通 Lesson | 前置补救课 |
|------|-----------|-----------|
| 目的 | 学习新知识 | 填补前置缺口 |
| 知识类型 | 混合（概念/原理/程序） | 仅概念性（What/Why） |
| 讲解策略 | 视知识类型而定 | **动机铺垫 → 核心定义 → 直觉类比** |
| 内容量 | 15-20 分钟阅读量 | **5-8 分钟，聚焦单一概念** |
| 习题数 | 3-5 题 | **2-3 题，记忆+理解层级为主** |
| 更新 roadmap | 是（主线进度） | **否（临时插入，不计入主线路线）** |
| 更新 session_state | 是（current_lesson 递增） | **否（仅记录，不影响主课编号）** |

### 多个前置缺口时：串行补救课

当递归依赖链展开后有多个前置知识点时，**每个概念单独生成一个极简补救课**，按依赖顺序串行插入：

```
诊断掌握度 43% → 检测到 3 个前置缺口
    ↓
lesson_4_prereq_electric_level.md（电平概念，5分钟）
    ↓ 完成后复诊
lesson_4_prereq_digital_circuit.md（数字电路基础，5分钟）
    ↓ 完成后复诊
lesson_4_prereq_clock_signal.md（时钟信号，5分钟）
    ↓ 完成后复诊
→ Lesson 4 主课（此时所有前置缺口已填补）
```

所有补救课**不更新 session_state.json 的 current_lesson**，主课的 current_lesson 保持不变。

### 快速回顾小节格式（错 1 题时用）

当掌握度在 33%-60% 之间，不单独开补救课，在**主课 Lesson 文件开头**插入：


## 大测驗模块（独立入口）

### 定位

大测驗是一个**可选的全面摸底工具**，在用户开始一个新专题时自动建议，也可由用户手动触发。

目的：一次性建立用户在某专题上的**完整知识图谱**，后续所有课程生成均基于此图谱进行难度适配。

### 触发时机

| 触发方式 | 条件 | 说明 |
|---------|------|------|
| 自动建议 | 首次开新专题（无知识图谱记录） | 非阻塞，建议用户做大测驗，但不强制 |
| 手动触发 | 用户说"大测驗"、"全面测试"、"做个摸底" | 随时可触发，重建知识图谱 |
| 定期复诊 | 间隔超过 14 天未学习该专题 | 建议用户重新测驗，刷新知识图谱 |

### 题目结构

```
100+ 题 = 约 35 个知识点 × 每点 3 题（L1+L2+L3）

每个知识点三层难度：
  L1（记忆）：名词解释、判断正误 — 确认基础概念存在
  L2（理解）：选择、填空、概念辨析 — 确认能正确解释
  L3（应用/分析）：分析时序、计算、写代码片段 — 确认能迁移使用
```

### 知识点覆盖清单（topic-knowledge-map.md）

为每个专题维护一个知识点树，标注核心/边缘分类：

```markdown

  - 开漏输出的意义 ★
  - 多主机仲裁基础 ○（边缘，自动生成）

### 协议层
  - 起始与停止信号 ★
  - 数据有效性 ★
  - 7位地址与R/W位 ★
  - ACK/NACK 机制 ★
  - 复合读写时序 ★

## CAN 总线
## RS485 总线
```

### 分批答题与进度保存

大测驗分多批完成，每批 20-25 题（预计 15-20 分钟）：

```

    ↓ 用户下次继续
系统读取 progress.json → 继续下一批
```
